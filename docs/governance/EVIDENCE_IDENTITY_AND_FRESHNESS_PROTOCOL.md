# Evidence Identity and Freshness Protocol

## Document status

```text
Protocol version: orchestra-evidence-v1
Repository: Baelfyre/Orchestra
Issue: #195
Phase: The Tuner Phase 2
Authority: Evidence and continuity integration only
Typed Tuner runtime: NOT AUTHORIZED
Persistence, SQLite, and RPC: NOT AUTHORIZED
Release and deployment: NOT AUTHORIZED
```

This document is the canonical source of truth for deterministic evidence identity, evidence-to-contract binding, generated-artifact lifecycle evidence, and stale-evidence enforcement. It supplements the cross-specialist coordination and delegated-execution protocols without changing specialist, governance, implementation, transition, Git, external-action, release, or deployment authority.

## 1. Authority boundaries

- Overseer remains the validation strategy and evidence owner.
- Arbiter remains the continuity and transition-decision authority.
- The Tuner identifies stale coordination references and recommends minimal re-entry, but does not validate its own work or issue transitions.
- Conductor remains the exclusive router.
- Ponytail and other implementation roles report complete change identity but do not approve continuation.
- Evidence proves conformance to current state. It never creates authority.
- Dagger and external actions remain separately authorized and default-deny.

## 2. Applicability

The same identity semantics apply in manual mode and delegated mode.

Manual mode binds evidence to:

```text
manual authorization reference
repository and branch
approved baseline SHA
frozen cross-layer contract hash and revision
current commit and working-tree fingerprint
```

Delegated mode binds evidence to the same identity plus the active `DelegatedExecutionEnvelope` and approved internal unit.

Manual mode must not invent a delegated envelope. Delegated mode must not use current evidence to widen the envelope.

## 3. Canonicalization

Identity-bearing packet hashes use canonical UTF-8 JSON.

```text
canonicalization_version: orchestra-evidence-v1
hash_algorithm: sha256
encoding: UTF-8 without BOM
object keys: lexical order
JSON separators: compact
NaN and Infinity: prohibited
path form: repository-relative POSIX
trailing newline in hash input: none
timestamps in identity input: excluded
```

Semantic lists preserve their declared order. Set-like lists generated by the collector are sorted by normalized path or stable identifier before serialization. Unknown canonicalization versions fail closed.

## 4. Git clean-filter semantics

File identity must use Git clean-filter semantics so the same tracked content has the same identity across supported line endings and working-tree representations.

For a working file, the canonical blob identity is produced by:

```text
git hash-object --path=<repository-relative-path> -- <repository-relative-path>
```

Evidence records paths and hashes only. It must not embed file contents, secrets, credentials, private keys, tokens, populated environment files, or private user data.

Each file identity records:

```yaml
path: repository-relative POSIX path
file_kind: regular | symlink
git_object_format: sha1 | sha256
blob_oid: Git object identity
byte_length: working representation length
```

## 5. Tracked patch identity

`tracked_patch_hash` is SHA-256 over the exact stdout bytes of:

```text
git diff
  --binary
  --full-index
  --no-ext-diff
  --no-color
  --no-renames
  --src-prefix=a/
  --dst-prefix=b/
  <approved_base_sha>
  --
```

It represents all tracked changes relative to the approved baseline, including committed descendants, staged changes, and unstaged tracked changes.

## 6. Staged patch identity

`staged_patch_hash` is SHA-256 over the exact stdout bytes of:

```text
git diff
  --cached
  --binary
  --full-index
  --no-ext-diff
  --no-color
  --no-renames
  --src-prefix=a/
  --dst-prefix=b/
  HEAD
  --
```

The staged hash is valid only with the recorded `current_commit_sha`. A matching staged hash against another commit is not current evidence.

## 7. Untracked and added-file identity

Every non-ignored untracked file must appear in `untracked_file_manifest`.

```yaml
untracked_file_manifest:
  - path
    file_kind
    git_object_format
    blob_oid
    byte_length
```

Rules:

- entries sort by normalized path;
- binary, empty, Unicode-path, and nested files are supported;
- directories are represented by file entries;
- ignored files are excluded unless declared as relevant generated artifacts;
- omission of any non-ignored untracked file invalidates evidence.

Every file added between the approved baseline and current commit, or added in the index, must appear in `added_blob_hashes`.

```yaml
added_blob_hashes:
  - path
    git_object_format
    blob_oid
```

A file-list-only handoff is insufficient.

## 8. Working-tree fingerprint

The working-tree fingerprint is SHA-256 over canonical JSON containing:

```yaml
repository_identity
branch
approved_base_sha
current_commit_sha
tracked_patch_hash
staged_patch_hash
untracked_manifest_hash
added_blob_hashes_hash
relevant_artifact_lifecycle_hash
```

Timestamps are not identity-bearing. A fingerprint mismatch means evidence is stale.

## 9. ExecutionEvidencePacket extensions

The evidence packet adds:

```yaml
identity_canonicalization_version
collaboration_session_id
cross_layer_contract_hash
cross_layer_contract_revision
tracked_patch_hash
staged_patch_hash
untracked_file_manifest
added_blob_hashes
artifact_lifecycle_records
specialist_reentry_completed
open_invalidation_events
```

Required freshness references are:

```yaml
repository_identity
branch
approved_base_sha
current_commit_sha
working_tree_fingerprint
cross_layer_contract_hash
cross_layer_contract_revision
```

Missing, malformed, unknown, or mismatched identity is blocking evidence failure.

## 10. GeneratedArtifactLifecycleRecord

Each relevant generated artifact records:

```yaml
artifact_id
collaboration_session_id
contract_packet_revision
producer_role
producing_command
path
artifact_type
state_before: ABSENT | PRESENT_TRACKED | PRESENT_UNTRACKED | PRESENT_IGNORED
state_after
content_hash
inspection_required
inspection_owner
inspection_completed
retention_required
cleanup_authority
cleanup_condition
cleanup_performed
cleanup_evidence
freshness_binding:
  baseline_sha
  current_commit_sha
  contract_hash
```

Pre-existing artifacts must not be deleted merely because a current command reuses the same path. New temporary artifacts may be cleaned only when cleanup authority and the cleanup condition are declared. Retained evidence artifacts must not be cleaned. Source-only validation cannot satisfy required generated-artifact inspection.

Common artifact classes include `dist/`, `test-results/`, `playwright-report/`, `coverage/`, `.wrangler/`, `.cloudflare-build/`, Python caches, test caches, adapter exports, validation reports, and generated diagrams when material to the current contract.

## 11. InvalidationEvent freshness

An `InvalidationEvent` identifies the exact trigger and affected contracts, evidence, artifacts, documentation, and diagrams.

```yaml
status: OPEN | REENTRY_ROUTED | REVALIDATED | RESOLVED
trigger_source
trigger_category
affected_contracts
affected_evidence
affected_artifacts
affected_docs_and_diagrams
```

Rules:

- specialist-declared invalidation dependencies are authoritative;
- The Tuner traverses only declared dependencies;
- only affected specialists are recommended for re-entry;
- Conductor routes the re-entry set;
- Overseer replaces stale evidence after contract refresh;
- Arbiter blocks continuation while required events remain open;
- unknown dependency edges fail closed without inventing a domain decision.

## 12. Arbiter freshness enforcement

Arbiter must return `HOLD`, `WAIT_FOR_EVIDENCE`, or a higher-priority disposition when any of the following is true:

- the packet is not `FROZEN`;
- the contract hash or revision is stale;
- branch, current commit, or approved baseline differs;
- tracked patch hash differs;
- staged patch hash differs;
- the untracked manifest is incomplete;
- added-file identities are incomplete;
- the working-tree fingerprint differs;
- a required artifact lifecycle record or inspection is missing;
- an invalidation event is open;
- specialist re-entry is incomplete;
- identity canonicalization is unknown or malformed.

`STOP` and `ESCALATE_HUMAN` retain higher transition precedence. Arbiter does not determine domain correctness.

## 13. Overseer evidence binding

Overseer must:

- derive validation requirements from the frozen contract acceptance matrix;
- bind evidence to the current contract hash and revision;
- collect or verify current change identity;
- inspect required generated artifacts;
- disclose skipped checks and limitations;
- reject source-only evidence when generated output inspection is required;
- produce evidence without issuing a transition disposition.

## 14. Implementation handoff

Ponytail or the active implementation role must produce a complete `SpecialistHandoffDelta` containing tracked, staged, untracked, and ignored-relevant paths; patch and blob identities; behavioral deltas; invalidated contracts; artifact deltas; required re-entry; and known limitations.

The Tuner compares the handoff with the frozen packet, opens explicit invalidation events, marks affected evidence and artifacts stale, and recommends the minimal re-entry set. It never dispatches specialists or resolves a domain conflict.

## 15. Fail-closed rules

The following must never default to readiness:

- unknown packet or coordination status;
- unsupported canonicalization version;
- Git command failure;
- unsafe or escaping path;
- omitted untracked or added file;
- mismatched branch, baseline, commit, patch, blob, artifact, or contract identity;
- open contradiction or invalidation;
- incomplete specialist re-entry;
- cleanup without authority;
- evidence containing secret material rather than hashes and paths.

## 16. Schema and independent verification clarifications

- Identity, safety, authorization, cleanup, inspection, and readiness flags require exact JSON booleans. Strings, numbers, nulls, objects, arrays, and missing values fail closed.
- Re-entry triggers, dependency edges, and owners require non-empty declared identifiers. An unknown trigger or missing owner blocks re-entry computation instead of producing an empty route.
- Windows drive-qualified, UNC, device, POSIX absolute, traversal, and malformed repository paths are prohibited.
- Repository identity must remove URL user information and credentials. Local paths are represented only by non-reversible hashes.
- Unexpected Git command failures are blocking. Expected detached-HEAD, absent-origin, and ignored-file states must be distinguished by return code rather than empty output.
- The approved baseline must be supplied explicitly or derived from an explicit verified CI base reference. `HEAD` is not an implicit baseline.
- Artifact evidence must be validated against independent artifact lifecycle records and current artifact state. A document cannot validate artifact claims sourced only from itself.
- Relevant present artifacts require verified SHA-256 content identity, complete lifecycle fields, and freshness binding to the approved baseline, current commit, and contract hash.

## 17. Read-only collector boundary

`scripts/evidence_identity.py` and its validators may read repository state and execute read-only Git commands. They may not stage, restore, reset, clean, remove, commit, amend, push, fetch external content, open pull requests, merge, release, deploy, or modify external systems.

## 18. Phase boundary

Phase 2 provides deterministic evidence and continuity enforcement at the protocol, output-contract, validator, and behavior-test layers. It does not add typed Tuner runtime models, persistent state, SQLite, RPC, host orchestration, release automation, or deployment behavior.
