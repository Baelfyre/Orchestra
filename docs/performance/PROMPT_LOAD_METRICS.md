# Prompt Load Metrics

## Purpose
This document explains how Orchestra measures prompt payload efficiency for the router-first architecture using a lightweight approximation script.

## Scope
The measurement method targets contextual markdown files actively used by the Conductor during intent classification, mode selection, and governance routing.

## Measurement Method
The script `scripts/measure_prompt_load.py` parses specified repository paths and calculates:
- Line Count
- Word Count
- Character Count
- Approximate Token Estimate

## Token Estimate Disclaimer
The token counts generated by this script are **rough approximations** based on the heuristic `token_count = character_count / 4`. This does not reflect exact BPE or WordPiece tokenizer results. It is meant exclusively as a lightweight comparative metric to track relative payload size reduction.

## Context Groups
Files are measured across four groups to highlight context selection filtering:
1. **Group A: Core Router-First Minimal Context**: The only files loaded by default.
2. **Group B: Broader Routing Context**: Loaded only during ambiguous routing resolution.
3. **Group C: Governance Context**: Loaded only during GOVERNED/AUDIT modes or when triggered.
4. **Group D: Baseline Performance Docs**: Measured for tracking documentation overhead.

## How to Run
```bash
python scripts/measure_prompt_load.py
```

## Expected Output
The script prints per-file metrics, per-group totals, and a grand total, highlighting how small Group A is compared to the grand total.

## Interpretation Rules
- If Group A's estimated token count grows significantly, the `ROUTER_FIRST_ARCHITECTURE.md` principle is being violated.
- Group C should remain heavy as it contains deep compliance rules, which is why it is excluded from FAST/STANDARD execution.
- A successful router-first strategy is proven when the average default execution prompt loads only Group A.

## Baseline Comparison Use
This metric script directly proves the benchmarks defined in `ROUTER_VALIDATION_BENCHMARKS.md`, verifying the reduction from "Very High" to "Low" prompt load.

## CI Integration
Prompt load metrics are measured automatically in the `governance-check.yml` CI workflow. The script output is captured and published as part of the `governance-validation-report` CI artifact (`artifacts/prompt_load_metrics.txt`). CI also executes the threshold checker (`artifacts/prompt_load_threshold_report.txt`), publishing both raw prompt load metrics and threshold status. Currently, these metrics are for observability only and do not trigger hard CI failures based on prompt size thresholds.

For a complete guide to accessing and interpreting all governance artifacts, see the [CI Artifact Index](../testing/CI_ARTIFACT_INDEX.md).

## Threshold Policy
For details on the acceptable growth limits of prompt payload sizes and the actions required when these limits are exceeded, see [PROMPT_LOAD_THRESHOLD_POLICY.md](PROMPT_LOAD_THRESHOLD_POLICY.md).

## Non-Goals
This method does not measure user prompt sizes, session history sizes, or downstream specialist output token generation costs.
