# Orchestra v1.1.2: Trusted Runtime Authority

`v1.1.2` is prepared for maintainer review. The current public release remains `v1.1.1`; no `v1.1.2` tag or GitHub Release is created by this release-preparation branch.

## Release Theme

This patch completes Orchestra's trusted runtime authority line. It turns the approved authority, runtime capability, bounded delegation, and lifecycle contracts into merged runtime behavior with fail-closed enforcement and deterministic audit evidence.

## Completed Scope

- Trusted authority is loaded from repository-controlled policy and evaluated as an explicit finite authority mode.
- Immutable run-scoped capability manifests are resolved before governance and cannot be widened by prompts, adapter context, routes, or governance decisions.
- Delegated child runs are bounded by parent authority, capabilities, depth, specialist identity, and explicit context inheritance.
- Lifecycle control uses structured signals and exact state transitions, with deterministic replay and conflict handling.
- `RuntimeExecutor` initializes each root or child run identity once and applies authority and capability checks before governance and adapter execution.
- Adversarial validation covers initialization, escalation, provenance, binding ownership, delegation, lifecycle, execution ordering, replay, and audit-sink failure paths.
- Audit events remain deterministic evidence. They do not grant authority or capabilities.
- Four governed Artificer promotions are finalized as `IMPLEMENTED`, with the Pattern Catalog synchronized manually.

## Promotion and Attribution Boundaries

The four promotion records retain pinned provenance, Apache-2.0 attribution language, conceptual-adaptation limits, and `automatic_promotion: false`. No external source code, prompts, payloads, examples, media, or documentation expression is included in this release preparation.

## Compatibility and Adapter Boundaries

Codex, Claude Code, and Antigravity are supported integration surfaces. Cursor, Windsurf, VS Code/VSCodium, JetBrains, Zed, and Neovim retain canonical scaffold-only maturity. Version normalization does not represent marketplace publication or adapter graduation.

Compatibility authority is explicit and finite. PRAP adapter capabilities remain separate from runtime authority and run-scoped capability grants.
